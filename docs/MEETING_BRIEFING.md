## 1) Цель встречи (ожидаемый результат)

1. Показать работоспособный end‑to‑end прототип (OWS v3 → storage → analytics → agent → UI).
2. Продемонстрировать соответствие ключевым требованиям ТЗ:
   - проверяемость (цифры воспроизводимы, есть evidence IDs/ссылки);
   - explainability (N, метод, ограничения);
   - выявление аномалий (цена/объем);
   - инкрементальность (ежедневное обновление < 24ч).
3. Зафиксировать текущие ограничения качества данных и план закрытия gaps (без «маскировки» реальности данных).

---

## 2) Короткое резюме (30–60 секунд)

Реализована аналитическая система, которая загружает данные госзакупок РК из OWS v3 в нормализованное хранилище PostgreSQL (с FK‑целостностью и lineage), строит аналитические витрины в DuckDB (Fair Price, аномалии цены/объема), и предоставляет строгий AI‑агентный слой. Агент не генерирует SQL и не вычисляет числа: он вызывает только allow‑listed инструменты с фиксированными шаблонами запросов и возвращает структурированный ответ с доказательствами.

## 3) Что показывать на демо (обязательные классы из ТЗ)

### 3.1 Демо‑интерфейс и API

- UI (Streamlit): `http://127.0.0.1:8501`
- Agent API (FastAPI): `http://127.0.0.1:8002/docs`
- Analytics API (FastAPI): `http://127.0.0.1:8001/docs`

### 3.2 Три обязательных кейса (готовые формулировки)

1) **Справедливость цены (Fair Price)**
- Пример вопроса:  
  `Оцени адекватность цены лота № 32899983 относительно аналогичных контрактов других ведомств в идентичном городе поставки`
- Что показать:
  - L0 вердикт + N + период + scope/fallback.
  - Top‑K сопоставимых контрактов (ID + ссылки).
  - Ограничения (например, если `region_id=NULL`, будет национальный fallback).

2) **Аномалии цен (>30%)**
- Пример вопроса:  
  `Найди закупки с отклонением цены >30% по ЕНСТРУ 331312.100.000000 за 2024 год`
- Что показать:
  - список Top‑K аномалий;
  - отклонение (%), ref_n, fallback_level;
  - обогащение (заказчик/поставщик, ссылки на контракт/объявление).

3) **Аномалии объема (YoY)**
- Пример вопроса:  
  `Выяви нетипичное завышение количества ТРУ по сравнению с предыдущими годами по ЕНСТРУ 172213.000.000002 за 2025 год`
- Что показать:
  - ratio, текущий объем vs предыдущий год;
  - customer_bin (и, по возможности, название заказчика);
  - ссылку на plan/view (evidence для объема).

### 3.3 Дополнительный кейс (для техкоманды): audit trail

- Пример вопроса:  
  `История договора №24764966`
- Что показать:
  - цепочку доступного provenance: plan → announcement → lot → contract → items (по мере заполненности данных);
  - принципы построения цепочки (не всегда есть прямой `contract -> lot` в OWS).

Ссылки на демо‑кейсы и JSON‑снапшоты: [`docs/DEMO_CASES.md`](DEMO_CASES.md), `artifacts/demo_cases/*.json`.

## 4) Ключевые инженерные решения (коротко, без «воды»)

### 4.1 Архитектура (API → storage → marts → agent)

```text
OWS v3 REST
  -> ETL loaders (resume/idempotent/FK-safe)
  -> PostgreSQL (source-of-truth, lineage, raw landing)
  -> Parquet export (incremental)
  -> DuckDB marts (market stats, fair price eval, anomalies)
  -> Analytics API + Agent API (strict tools, no free-form SQL)
  -> Streamlit UI (demo)
```

### 4.2 Хранилище и lineage

- Нормализованная схема PostgreSQL + внешние ключи.
- Критическая связь plan→lot сохраняется через `lot_plan_points` (иначе теряется audit trail).
- Для предотвращения FK‑ошибок внедрены `ensure_enstru_ref` и `ensure_subject` (см. `etl/utils.py`).

### 4.3 Производительность и устойчивость ingestion

- Resume/checkpoint (`etl_state`) + конечный backstep (`RESUME_BACKSTEP`).
- Adaptive RPS (снижение на 429/5xx) + `--concurrency`.
- In‑flight dedupe на повторяющиеся lookups (например, `trd-buy/{id}`).

### 4.4 Аналитический слой (deterministic)

- Fair Price: median baseline + статистика IQR; time adjustment через `macro_indices` (CPI index).
- Аномалии цены: пороговое правило `|deviation_pct| > 30%` на витрине `lot_fair_price_eval`.
- Аномалии объема: YoY ratio по `volume_anomalies`.
- Ключевой принцип: «LLM не считает и не формирует цифры».

### 4.5 Агентный контур (безопасность + воспроизводимость)

- Intent → параметризованный tool из whitelist → structured JSON → шаблон ответа.
- Запрещено: генерация SQL, доступ к сырым таблицам, «придумывание» чисел.
- Наличие регрессионных тестов intent‑маршрутизации: `tests/test_canonical.py`.

## 6) Известные ограничения (проговорить заранее) и план закрытия

### 6.1 Ограничения (текущая реальность данных)

- Высокая доля `NULL` в региональных атрибутах по части сущностей.
- Ограниченная полнота некоторых полей в `lots` (например, `enstru_code`) вследствие структуры OWS payload и текущей стратегии извлечения.
- Объем `contract_items` на текущем срезе ниже целевого уровня для статистически сильной региональной Fair Price.

### 6.2 План закрытия (в инженерных терминах)

- Расширение покрытия `contract_items` через повторяемые resume‑прогоны `etl/load_contract_items.py` с `--concurrency` и backstep (идемпотентно).
- Повышение регионального покрытия: продолжение checkpointed воркера `/v3/plans/kato` (`etl/load_plans_kato_incremental.py`) + repair‑циклы.
- Улучшение ENSTRU в лотах: приоритетная ориентация на прямые поля OWS payload (если доступны) + сохранение fallback на plan_points.
- Жесткое соблюдение quality gates (см. [`docs/EXECUTION_CHECKLIST.md`](EXECUTION_CHECKLIST.md)).

## 7) Вопросы, которые ожидаемо возникнут (и рекомендуемые ответы)

### 7.1 Руководитель нац.проектов / big data (фокус: риски, воспроизводимость, on‑prem)

1) **Как доказывается корректность цифр?**  
Ответ: все числа считаются в витринах DuckDB/SQL, в ответе возвращаются evidence IDs и ссылки; integrity проверяется скриптами `verify_spiral2.py`/`verify_spiral3.py`.

2) **Как обеспечивается обновление < 24 часов?**  
Ответ: предусмотрен инкрементальный ingestion по `/v3/journal` и ежедневный запуск `scripts/run_daily.sh`; прогресс хранится в `etl_state`.

3) **Готовность к on‑prem / безопасность?**  
Ответ: Docker Compose, порты на `127.0.0.1`, изолированная сеть `data_net`, секреты в `.env` (не коммитятся), LLM не получает сырые данные.

### 7.2 Руководитель отдела ИИ (фокус: agentic workflow, hallucination control, метрики)

1) **Почему LLM не считает и не пишет SQL?**  
Ответ: требование проверяемости. Агент вызывает только whitelist tools, возвращающие структурированный JSON. Это снимает риск hallucination и SQL‑инъекций.

2) **Как будет достигаться KPI 85%?**  
Ответ: метрика трактуется как точность извлечения параметров + точность чисел ответа относительно tool‑результатов; предусмотрен labeled set и регрессионные тесты.

### 7.3 Разработчики/ML‑специалисты (фокус: данные, алгоритмы, масштабирование)

1) **Почему DuckDB, а не сразу ClickHouse?**  
Ответ: DuckDB дает быстрый OLAP слой и упрощает витрины при минимальной инфраструктуре; при росте объема возможна миграция витрин в ClickHouse без изменения агентного контракта.

2) **Как реализована Fair Price?**  
Ответ: median baseline + IQR/MAD‑style статистика; time adjustment через CPI index; fallback ladder (region/year) и вердикт `insufficient_data` при N < 20.

3) **Как обрабатываются 500/429 в OWS?**  
Ответ: retry/backoff + adaptive RPS; прогресс сохраняется через checkpoint, поэтому процесс не требует рестарта «с нуля».
